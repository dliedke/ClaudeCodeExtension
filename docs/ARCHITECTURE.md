# Architecture — Key Non-Obvious Details

> On-demand reference. Read the relevant section **before editing the file it describes**
> (see the index in `CLAUDE.md` → Architecture). These are the non-obvious gotchas and
> design decisions that aren't apparent from the code alone.

## Terminal Embedding (Terminal.cs)

- **Two terminal modes**: Command Prompt (conhost) and Windows Terminal (wt.exe), via `_settings.SelectedTerminalType`
- **Lifecycle serialization**: `_terminalLifecycleSemaphore` prevents overlapping start/stop transitions. On top of it, `_terminalLaunchTicket` (monotonic, incremented per `StartEmbeddedTerminalAsync` call) makes queued duplicate launches collapse: a launch that finds a newer ticket after acquiring the semaphore skips itself ("skipping superseded launch request" in the launch log), and the relaunch-retry loop aborts its remaining backoff attempts when a newer request queues up behind it (no MessageBox in that case). Without this, every queued launch tore down the terminal the previous one had just embedded — WSL stop/start churn that feeds the issue #73 failure modes
- **Session ID tracking**: `_terminalStartupSessionId` discards stale startup work when a new terminal start is triggered before the old one finishes
- **`SetParent()` retry**: Up to 3 attempts with 200ms delay and Win32 error logging (`Marshal.GetLastWin32Error()`), re-applies window styles between retries
- **Conhost handle discovery**: `FindMainWindowHandleByConhostAsync()` retries with 5s then 10s timeouts; uses ToolHelp32 (`CreateToolhelp32Snapshot`) for child PID lookup. Accepts only `ConsoleWindowClass` windows — a PID match alone can hit other top-level windows reporting the same PIDs (e.g. the conhost thread's invisible "Default IME" window), and embedding one of those leaves the panel blank while the real console floats (issue #73). Aborts early when the launcher process has exited
- **Unified relaunch retry + launch log** (issue #73): the conhost path in `StartEmbeddedTerminalAsync` wraps the entire spawn→find→embed unit in a bounded relaunch loop (`relaunchBackoffMs = {0, 750, 1500, 3000, 6000}`, 5 attempts since 10.98 — a 10.97 log showed WSL teardown contention outlasting the original ~2s ladder by minutes). Reproduced on the author's own machine 2026-06-11: two transient, **WSL-only** failure modes that clear within ~1 min, both triggered by switching solutions/providers while the just-stopped `wsl bash` session is still tearing down — (A) conhost spawns then exits immediately with `exitCode=0` (no window), and (B) the window is found but the conhost dies mid-embed so `SetParent` fails repeatedly with Win32 error **1400** (`ERROR_INVALID_WINDOW_HANDLE`). Both now relaunch instead of giving up: between attempts `StopExistingTerminalAsync()` kills the orphan window/process (so retries don't pile up live conhosts) then backs off. The inner spawn loop still re-spawns once on immediate exit; no inner retry when the launcher is alive but windowless (would pile up live terminals — that case defers to the outer relaunch via cleanup). Console color/font registry is saved **once before** the loop and restored **once after** (re-saving per attempt would clobber the originals). Only after all relaunch attempts fail does a MessageBox appear. Launch steps (teardown, spawn PID, whether VS had a console attached, find/SetParent failures with Win32 errors, exit codes) append to `%LocalAppData%\ClaudeCodeExtension\terminal-launch.log` via `LogTerminalLaunch()` (best-effort, never throws, resets at 512 KB) — `Debug.WriteLine` is compiled out of Release builds, so this log is the only way to diagnose user machines; ask reporters to attach it
- **Embed failure honesty + panel self-repair** (issue #73): both embed paths verify `SetParent` actually took (`SetParent` returning zero is ambiguous — a top-level window's previous parent is also zero, so success is confirmed via `GetParent`) instead of falling through claiming success (which left a hidden, parentless window = blank panel). The conhost path re-checks `IsWindow(terminalHandle)` + `HasTerminalProcessExited(cmdProcess)` **before each `SetParent`** so a window that died mid-embed (mode B above, Win32 1400) is detected at once and deferred to the outer relaunch rather than burning 3×200ms retrying against a corpse; a genuine embed failure also defers to the relaunch loop, and only the final post-loop failure reports via MessageBox + launch log. `EnsureTerminalPanelReady()` recreates `terminalPanel` and re-hosts it in `TerminalHost` when it is missing/disposed — previously a dead panel made the launch silently bail, and since the dead panel persisted, *every* later restart stayed blank until the project was reopened (the only failure mode matching that exact symptom)
- **Teardown wait — all providers** (issue #73; WSL-only in 10.98, extended in 11.0): after killing the process trees, `StopExistingTerminalAsync` polls the killed PIDs for up to 3s until they actually exit ("waited Nms for terminal teardown" in the launch log). Spawning the next terminal while the old session is still dying is what makes the fresh conhost exit instantly with code 0 or lose its window mid-embed — first reproduced with WSL teardown, but a reporter's log showed the **same fingerprint with native Claude Code** (slow claude/node tree teardown, e.g. corporate EDR), which is why the wait is no longer WSL-gated. Three supporting pieces (11.0): `KillProcessAndChildren` records **every** PID it kills into the caller's set (`Process.Kill` only *initiates* termination, and the old set held only tree roots — a lingering grandchild like the agent's node process let the wait pass early); the per-launch `_pendingTeardownProcessIds` set carries those PIDs into the relaunch loop, which holds each respawn up to 5s more while any are alive (an instant-exit attempt leaves nothing of its own to wait on, so without this the ladder kept respawning into the same dying tree); and the "conhost exited immediately" log line appends `oldSessionProcsAlive=N` so a user log alone confirms or rules out teardown contention
- **WT embedding**: Finds `CASCADIA_HOSTING_WINDOW_CLASS`, embeds with `WS_CHILD`, calculates tab bar height offset; the WT path logs to the launch log like the conhost path (spawn PID, window-not-found, SetParent failures)
- **Terminal hidden from taskbar**: `WS_EX_TOOLWINDOW` + clear `WS_EX_APPWINDOW`
- **F5 forwarding**: Low-level keyboard hook (`WH_KEYBOARD_LL`) intercepts F5/Ctrl+F5/Shift+F5 → VS debug commands via DTE
- **Mouse hook** (`WH_MOUSE_LL`): Tracks Ctrl+Scroll zoom delta (persisted); converts plain left-drag to SHIFT+drag for WT text selection. **Conhost Ctrl+Scroll zoom is input-mode-independent** (issue #76/#78): for the Command Prompt terminal (`_wtTabBarHeight == 0`), `TryForwardConhostCtrlZoom` runs on the hook thread (Win32 `GetWindowRect` hit test, no WPF/WinForms), consumes the physical wheel (returns 1), and offloads the actual zoom to `Task.Run(() => TryAdjustConhostFontSize(±1))` which changes the console **font size directly** via `SetCurrentConsoleFontEx` (then persists the delta + schedules the repaint on the UI thread). The earlier v19 approach of posting `WM_MOUSEWHEEL + MK_CONTROL` to the conhost window did **not** work once a TUI put the console into mouse-input mode (QuickEdit off): conhost forwards both the physical wheel **and** a posted wheel to the app as input records, so neither reaches conhost's own zoom — that's why zoom worked only before the agent's TUI came up. Setting the font is independent of console input mode, so it works whether the TUI is up or not; the saved-zoom replay in `ApplyTerminalZoomDeltaAsync` uses the same `TryAdjustConhostFontSize(delta)` for the conhost path so interactive and replay zoom stay consistent. WT is left on its native path (dispatch → `HandleMouseHookEvent`). **Right-click is no longer intercepted at all** (v29.0, root-fix for issue #90): the hook does **not** touch `WM_RBUTTONDOWN`/`WM_RBUTTONUP` — conhost/Windows handle the right mouse button natively everywhere (context menus in the editor/Solution Explorer, conhost's own QuickEdit-on right-click paste). The earlier "conhost right-click paste fallback" (issues #78/#82/#83) consumed right-clicks via a global `WH_MOUSE_LL` hook to paste into the agent's TUI when QuickEdit was off; despite an occlusion-aware + foreground-gated hit test it kept mis-firing — a right-click elsewhere in VS would be swallowed and the clipboard pushed into the console (issue #90) — so the whole feature was removed (`TryConhostRightClickPaste`, `IsScreenPointOverTerminalWindow`, `_consumeNextRButtonUp`, `_conhostInMouseInputMode` cache + `QueueConhostMouseInputModeRefresh`, and the `WM_RBUTTONDOWN/UP` constants are all gone). Pasting into the agent goes through the **prompt box**, whose send path delivers via `PasteViaConhostPasteCommandAsync` (conhost's own Edit→Paste `WM_COMMAND/ID_CONSOLE_PASTE` posted straight to `terminalHandle`) regardless of QuickEdit/mouse-input mode — so no terminal right-click paste is needed. `PasteViaConhostPasteCommandAsync` and `IsTerminalInMouseInputMode()` remain, now used only by the send path (the mouse-input-mode probe still drives whether the send path injects its pre-deselect right-click). **Ctrl+Scroll zoom hijacked other apps when VS wasn't foreground (v51.0)**: both zoom paths — `TryForwardConhostCtrlZoom` (conhost) and the `ctrlDown` branch of `HandleMouseHookEvent` (Windows Terminal) — only hit-tested the cursor against the terminal's screen rectangle, which stays valid even when another maximized window is now layered on top of that same screen region (VS backgrounded). Global `WH_MOUSE_LL` doesn't know z-order, so Ctrl+Scroll in that other app got silently redirected into zooming the (invisible) embedded terminal instead of reaching the app the user was actually looking at. Fixed by extracting `IsTerminalHostForeground()` (the `GetForegroundWindow() == terminalHandle || hostRoot` check, previously inlined at the top of `IsTerminalFocused()`) and requiring it in both zoom paths before the rect hit test runs — cheap enough for the hook thread (same GetForegroundWindow-based short-circuit `IsTerminalFocused()` already relied on). Deliberately stops short of requiring full `IsTerminalFocused()` (keyboard focus on the terminal control specifically) so hover-to-zoom while VS is foreground still works without first clicking into the terminal.
- **Post-startup**: `SchedulePostStartupTerminalAdjustments()` runs deferred resize + zoom replay; `SchedulePostSolutionLoadTerminalRefresh()` does 200/500/1000ms repaint passes after solution load
- **Fresh PATH from registry**: `GetFreshPathFromRegistry()` reads PATH from `HKLM` and `HKCU` registry keys to detect newly installed tools (e.g. Windows Terminal) without requiring VS restart
- **WPF keyboard-focus reclaim** (issue #65): `SetParent`ing the terminal (separate-process window) into VS joins that process's input queue with the VS UI thread, so keyboard focus is shared state. If it gets stuck on the terminal, WPF's `Focus()` won't always move native focus back — the prompt stops accepting typing (caret stops blinking) and the provider menu can't be arrow-navigated until VS restarts. `ClaudeCodeControl_PreviewMouseLeftButtonDown` (tunneling, fires before any child handles the click) calls `ReclaimWpfKeyboardFocusIfStuck()`: when `GetFocus() != HwndSource.Handle` it `SetFocus`es the WPF host, so one click anywhere restores typing. No-op when WPF already owns focus
- **Terminal-click activation** (issue #69): a click on the embedded terminal runs `ActivateEmbeddedTerminalOnClick` (from the mouse hook), which raises the VS top-level window (`BringVisualStudioToForegroundIfNeeded`, AttachThreadInput dance to bypass focus-stealing protection) and selects the terminal pane (`IVsWindowFrame.Show()` when `IsTerminalToolWindowActive()` is false). **Do not re-attempt a "don't bring VS to the foreground on terminal click" opt-out** — it shipped in 10.82–10.90 and was removed in 10.91 as infeasible: keyboard input only goes to the foreground thread's focus window, so VS *must* be activated for typing to reach the terminal, and suppression attempts (`MA_NOACTIVATE` from the host panel, `WS_EX_NOACTIVATE` on the terminal — inert on `WS_CHILD` windows, gating `frame.Show()` via `ShowNoActivate()`) either didn't engage or would have broken typing. Windows itself also activates/raises the top-level owner when the cross-process child is clicked, outside the extension's control.
- **WPF-prompt focus guard** (v27.0, symmetric with the terminal guard): clicking the WPF prompt previously got only a single-shot `ReclaimWpfKeyboardFocusIfStuck()`, so a deferred steal (the cross-process terminal grabbing the shared native focus, or the OAF console attach on Command Prompt) landed a moment later and typing into the prompt went nowhere until the user clicked again. A **self-extending** guard now runs: `EnsureWpfPromptFocusGuardRunning()` (called from `ClaudeCodeControl_PreviewMouseLeftButtonDown` and `PromptTextBox_PreviewKeyDown`) refreshes `_lastPromptInteractionUtc` and, via `RunWpfPromptFocusGuardAsync`, re-asserts focus every 80ms until ~2s after the last prompt interaction (so it holds through active typing during generation, not just after one click). Single loop at a time (`_promptFocusGuardRunning`). It re-asserts **only when the terminal specifically holds native focus** (`GetFocus()` is the terminal or its child) — a WPF ContextMenu/Popup (the ⚙ menu) or child editor is left alone so menu/arrow nav isn't broken. Aborts on a terminal click (`_promptFocusGuardSeq`, bumped in `ActivateEmbeddedTerminalOnClick`) or an app switch, so the two guards never fight. **v28.0 de-escalation**: this guard window was cut to ~0.7s (`idleWindowMs`) and the terminal click guard to ~0.6s (`totalChecks = 8`). v27 had stretched the terminal guard to ~5s (to outlast the OAF console-attach focus bounce); with `FocusTerminalWindow` no longer merging an unrelated foreground thread's input queue (see below) the long windows were both unnecessary and harmful — they kept re-asserting focus for seconds and ping-ponged it between the terminal and the prompt, dropping keyboard/mouse input in the terminal while the agent was busy
- **Single-click terminal focus** (issue #74): the click natively focuses the terminal child, but both activation steps above make VS move keyboard focus into the WPF tool-window content moments later (`frame.Show()` focuses the pane content; `SetForegroundWindow` restores VS's last focused control) — so a click on the terminal could get its focus silently stolen before typing (e.g. answering an agent's choice prompt) reached it. The 10.94 fix re-asserted focus exactly twice at fixed +80/+160 ms — under heavy load (agent crunching, VS repainting) VS's restore landed *after* both asserts, so **every** click lost the race and the terminal became impossible to select (the issue's follow-up report). Since 11.0, `ActivateEmbeddedTerminalOnClick` always schedules `EnsureTerminalFocusAfterClickAsync`: 80 ms checks that re-assert via `FocusTerminalPanel` whenever `GetFocus()` isn't the terminal (or a child of it). It aborts when a newer click occurs anywhere (`_terminalClickSequence`, bumped per WM_LBUTTONDOWN in the mouse hook), when the foreground window leaves the VS root, or when the terminal/panel dies — so it can't fight the user, only VS's deferred restores. Focus state is read via `IsTerminalFocused()` (GetGUIThreadInfo, same check the F5 hook uses); ordinary clicks cost one such check per tick, no asserts. **Self-extending guard (v46.0)**: the guard runs a `baseChecks`=20 (~1.6 s) window and then keeps going only while the user is *actively typing into the terminal* (`_lastTerminalKeyUtc` within `typingExtendMs`=700 ms), hard-capped at `absoluteMaxChecks`=100 (~8 s). This fixed the "can't type when the AI asks a question" hang: v28 had cut this to a fixed ~0.6 s (`totalChecks = 8`), so a VS focus-restore (or a pause before answering) landing after that short window stranded the terminal without focus and the reply went nowhere. Because it aborts on *any* click elsewhere (every left-click bumps `_terminalClickSequence` via the global mouse hook) and only self-extends while terminal keystrokes are actually arriving, it can't reintroduce the v26/v27 idle ping-pong (that came from the old ~5 s *unconditional* window plus the foreground-thread `AttachThreadInput` that no longer runs). **`FocusTerminalWindow` (v28.0): SetFocus only, no foreground-thread `AttachThreadInput`.** `SetParent` already joined the terminal thread's queue to VS, so a plain `SetFocus(terminalHandle)` moves native focus; only when that doesn't take does it briefly attach the *terminal* thread (already joined, so safe) and retry. v26/v27 also attached the current *foreground* thread's input queue on every focus set — merging an arbitrary busy thread into VS's synchronous input queue, which backed up all keyboard/mouse input while the conhost thread was rendering agent output (the reported "input freezes while the agent works, recovers when idle"). **The F5 keyboard hook tracks `_lastTerminalKeyUtc` on every terminal keystroke (v46.0)** so the self-extending click guard (and the OAF typing guard) know when the user is typing an answer into the terminal. This reverses the v28 `needFocus` gate that only computed `IsTerminalFocused()` for F5 or while a completion watch was active — with OAF off, terminal typing went untracked, which is why the answer-the-agent hang persisted without OAF. It stays cheap because `IsTerminalFocused()` short-circuits on `GetForegroundWindow` (the cross-process `GetGUIThreadInfo` runs only when a VS window is foreground, i.e. at human keyboard rates). `StartHookThread()` is also re-invoked on every terminal launch (idempotent) so a dead hook thread — which silently disables click activation and F5 forwarding — revives on the next start. The hit-test also accepts child windows of the terminal (`IsChild`), not just the exact handle.

- **TUI fullscreen env vars** (v31/v32, `GetClaudeCommand`): `_settings.ClaudeTuiFullscreen` (nullable) injects an env prefix ahead of the `claude` command — `true` → `CLAUDE_CODE_NO_FLICKER=1` **plus `CLAUDE_CODE_DISABLE_MOUSE=1`**; `false` → `CLAUDE_CODE_DISABLE_ALTERNATE_SCREEN=1` (classic renderer); `null` → nothing (CLI default). cmd.exe form is `set X=1 && set Y=1 && {cmd}`, WSL bash form is `X=1 Y=1 {cmd}`. **`CLAUDE_CODE_DISABLE_MOUSE` is mandatory whenever fullscreen is on** (issue #92): fullscreen rendering enables Claude Code's own terminal mouse tracking, which in the embedded conhost — where the extension already runs a global mouse hook and re-asserts focus — emits a continuous burst of mouse-input escape sequences that Claude's fast-input heuristic collapses into `[Pasted text +N lines]` bracketed-paste blocks, flooding the prompt on launch and on every restart. The extension supplies its own mouse zoom / click-to-focus / paste, so disabling Claude's mouse capture loses nothing while keeping the flicker-free draw. The one casualty is wheel scrolling: with Claude's mouse off, the wheel no longer pages the conversation (per Claude's docs, fullscreen + `DISABLE_MOUSE` keeps only the keyboard scroll keys `PgUp`/`PgDn`/`Ctrl+Home`/`Ctrl+End`, plus native terminal selection of the visible viewport). **Wheel scroll restored without re-enabling Claude's mouse** (v36, issue #96): the global mouse hook's `HandleMouseHookEvent` now, on a plain (no-Ctrl) wheel notch over the active panel when `IsClaudeFullscreenConhostActive()`, posts `VK_PRIOR`/`VK_NEXT` (PgUp/PgDn) keystrokes to the terminal via `SendScrollKeyToTerminal`. Two gotchas: (1) `ShouldDispatchMouseHookMessage` must let the plain wheel through — it otherwise forwards `WM_MOUSEWHEEL` only when Ctrl is held (zoom), so the plain-wheel case is gated on `IsClaudeFullscreenConhostActive()` there too, or the handler never runs. (2) `SendScrollKeyToTerminal` builds a real `lParam` (scan code from `MapVirtualKey` + extended-key bit) — conhost silently drops a posted key with a zero scan code instead of translating it to VT input (`ESC[5~`/`ESC[6~`). Posted straight to `terminalHandle` (focus-independent, like the Enter path); gated to conhost (`_wtTabBarHeight <= 0`) and the Claude Code / Claude Code (WSL) providers so it never hijacks conhost's own scrollback in the classic renderer or Windows Terminal's native wheel handling. **Smooth line scroll** (vs. the half-page PgUp default): for *native* Claude Code, `TryEnsureFullscreenLineScrollKeybinding()` merges `pageup`→`scroll:lineUp`, `pagedown`→`scroll:lineDown` (and `shift+pageup/pagedown`→`scroll:halfPage*` to preserve paging) into `%USERPROFILE%\.claude\keybindings.json` — those line-scroll actions are unbound by default, so the wheel had nothing finer than half-page to trigger. With the binding in place each notch sends one PgUp/PgDn (`GetFullscreenWheelScrollKeystrokeCount()` returns 1) for a one-line step; WSL Claude (config lives in the WSL filesystem) and any write failure fall back to a single half-page keystroke. The keybinding file is touched once per session, lazily on first wheel scroll, and merges without overwriting existing user bindings (`AddBindingIfAbsent`).

**Command patterns**:
```
Windows: cmd.exe /k chcp 65001 >nul && cd /d "{dir}" && ping localhost -n 3 >nul && cls && {command}
WSL:     cmd.exe /k chcp 65001 >nul && cls && wsl bash -lic "cd {wslPath} && {command}"
```

**Console font / codepage registry** (`SaveAndSetConsoleFontRegistry` / `RestoreConsoleFontRegistry`): conhost reads `HKCU\Console` once when a console window is created, so the font (`FaceName` + `FontFamily=54` TrueType) is written **before** spawning conhost and restored after it has started. `FaceName` comes from `_settings.ConsoleFontFaceName` (Settings → Terminal; default `"Cascadia Mono"`, empty falls back to it) so users on CJK/non-Latin systems can pick a font that renders their script — Cascadia Mono has no Han/Kana/Hangul glyphs, so without this they relied on conhost's font fallback. Changing the font relaunches the terminal. `CodePage=65001` is written to the **per-executable** subkey `Console\%SystemRoot%_System32_conhost.exe` (the parent `Console\CodePage` is ignored by conhost in practice); the original values are saved and the subkey is deleted on restore if we created it.

**Console font under Windows Terminal** (`EnsureWindowsTerminalFontProfile` / `GetWindowsTerminalFontProfileArg`): WT ignores `HKCU\Console` and `wt.exe` has no font flag, so the only lever is `settings.json`. The same `ConsoleFontFaceName` setting drives a **dedicated managed profile** ("Claude Code Extension", fixed GUID `{b9a3c7e2-…}`) carrying just `font.face`; the launch passes `-p "Claude Code Extension"` (the trailing `-- cmd.exe {cmd}` still overrides the profile's command line). The profile inherits colors/cursor/etc. from `profiles.defaults`. Blast radius is kept minimal: `settings.json` is touched **only** when a non-default font is selected (`"Cascadia Mono"`/blank → no `-p`, untouched file) and **only** when the profile is missing or its font differs. Handles both schemas (`profiles.list` array and legacy bare `profiles` array). Caveat: rewriting via Newtonsoft serialization drops the user's `//` comments — accepted because writes are rare and gated to non-default fonts. Any failure (no settings.json, parse/write error) returns null and the launch falls back to WT's default profile. The Settings picker (a searchable, live-preview ListBox of `Fonts.SystemFontFamilies`) is shared by both terminal types.

**WSL path conversion** (`ConvertToWslPath()`): `\\wsl.localhost\distro\path` → `/path`, `C:\...` → `/mnt/c/...`

**WSL shell mode**: Uses `bash -lic` (login + interactive) to load `.profile`/`.bash_profile` PATH entries — applies to all WSL providers (Claude Code WSL, Codex WSL, Cursor Agent WSL, Devin). WSL launch commands must go through `BuildWslLaunchCommand()` so the cwd is Bash-quoted before it is embedded in the outer `cmd.exe` command.

## Provider Detection (ProviderManagement.cs)

- **Caching**: `_providerCache` (5-min TTL), separate `_wslCache` for WSL install status; `_cacheLock` for synchronized access; `IsCacheValid()` checks expiry
- **Claude Code detection**: Two-tier — first checks native path (`%USERPROFILE%\.local\bin\claude.exe`), then falls back to `where claude` (PATHEXT-aware, finds both `claude.exe` from winget and `claude.cmd` from NPM)
- **WSL detection**: `bash -lc` (login shell) for `which` — avoids `.bashrc` noise; retries 2x with 8s/20s timeouts for cold boot
- **PI detection**: `IsPiAvailableAsync()` runs `cmd /c where pi` (3s timeout), available when exit code 0 and stdout non-empty; native Windows NPM tool (`@earendil-works/pi-coding-agent`), TUI — paste uses Shift+Right-click and `WM_CHAR` Enter like Open Code
- **Antigravity detection**: `IsAntigravityAvailableAsync()` runs `cmd /c where agy` (3s timeout, PATH refreshed from registry), available when exit code 0 and stdout non-empty; Google's native Windows agent (install `irm https://antigravity.google/cli/install.ps1 | iex` to `%LocalAppData%\agy`, launched with `agy`). Runs in conhost, `WM_CHAR` Enter like Claude Code. **Paste workaround**: Antigravity disables conhost `ENABLE_QUICK_EDIT_MODE` at startup (re-disables faster than `SetConsoleMode` can restore), so a plain right-click opens the context menu instead of pasting. Dedicated paste branch right-clicks to open the menu, then Down ×3 → Enter via `keybd_event` to land on Paste (menu order Mark/Copy[disabled]/Paste/…; arrow nav stops on disabled items; locale-agnostic). Delays generous (500ms after right-click, 150ms between keys) — rushing drops keys. Excluded from the `isCommandPrompt` cancel-selection right-click
- **Reasonix detection**: `IsReasonixAvailableAsync()` runs `cmd /c where reasonix` (3s timeout, PATH refreshed from registry), available when exit code 0 and stdout non-empty. DeepSeek-native coding-agent TUI installed via `npm i -g reasonix` (so it lands on PATH); native Windows conhost, modeled on Antigravity/PI. Launched with plain `reasonix` (no flags, no model arg — minimal scope). Paste uses the shared `WM_COMMAND` conhost paste; Enter is a single `WM_CHAR` like PI/Open Code; exit/update interrupts the TUI with CTRL+C then runs `npm i -g reasonix`. DeepSeek API key (`DEEPSEEK_API_KEY`) is the user's own concern. Optional CLI-path override supported but not required
- **Devin (native) detection**: `IsDevinNativeAvailableAsync()` runs `cmd /c where devin` (3s timeout, PATH refreshed from registry), available when exit code 0 and stdout non-empty. Native Windows port of the same `devin` CLI as Devin (WSL); install must be done from Windows Terminal (`irm https://static.devin.ai/cli/setup.ps1 | iex`) but the resulting `devin` runs in both Windows Terminal and conhost. Launched with plain `devin` (no flags). Because it is the same `devin` TUI as Devin, it shares the configurable model list (`_settings.DevinModels`) and the Devin model menu (`ModelContextMenu_Opened` treats `DevinNative` like `Devin`); model switches send `/model "<name>"` (quoted) to the running terminal. Enter uses the KEYDOWN/KEYUP path (`SendEnterKeyDownUp`) like Devin, paste stays on the clipboard path (excluded from clipboard-free per-char keystrokes like PI/Reasonix), and exit sends `exit`; **update** force-kills lingering `devin.exe` then runs the official updater via PowerShell (`taskkill /f /im devin.exe >nul 2>&1 & powershell -NoProfile -ExecutionPolicy Bypass -Command "irm https://cli.devin.ai/install.ps1 | iex"`) because `devin update` does not update — it only prints the install command, and the installer's `Copy-Item` over `%LOCALAPPDATA%\devin\cli\bin\devin.exe` fails with "Access is denied" while any `devin.exe` is still running (self-update lock). Not a WSL provider (no path conversion). Optional CLI-path override supported
- **Early-exit logic**: Stops retrying only when stdout has content (ignores stderr-only warnings)
- **Notification flags**: Static booleans (one per provider) ensure install pop-ups show only once per VS session
- **Active provider UI source** (v24.0): provider checkmarks, provider-specific flag visibility, model/usage menu visibility, tool-window captions, detached captions, and visible-agent "active" labels use `GetActiveOrSelectedProvider()` — `_currentRunningProvider` when a terminal is alive, falling back to `_settings.SelectedProvider` only before launch. Successful launches sync `_settings.SelectedProvider` in memory, while `SaveSettings()` still preserves volatile provider/model/effort fields from disk during normal operation for multi-instance isolation.
- **Model menus**: `ModelContextMenu_Opened()` toggles Claude items vs Devin items based on active provider. For Devin (`Devin` and `DevinNative`), the model entries are built dynamically from the user-configurable `_settings.DevinModels` list (`RebuildDevinModelMenuItems` inserts checkable items at the top of `ModelContextMenu`, tracked in `_dynamicDevinModelItems` for removal on the next open), followed by the static "Configure Models..." item; picking one sends `/model "<name>"`. "Configure Models..." opens the list editor (`ShowDevinModelsDialog`) to add/edit/remove/reorder names. `SelectedDevinModel` is the chosen name; `GetDevinModelDisplayName()` resolves it (falling back to the first entry)
- **Effort slider** (v42.0, Claude only): the Claude effort picker is a pill `Slider` (6 stops) hosted in a `StaysOpenOnClick` `MenuItem` (`EffortSliderMenuItem`) inside `ModelContextMenu`, styled via `EffortSliderStyle`/`EffortFillButtonStyle`/`EffortThumbStyle` in `ClaudeCodeControl.xaml`. Mirrors the VS Code effort slider; the visual order (Low, Medium, High, XHigh, Max, Ultracode) is defined by `_effortSliderOrder` and is **decoupled** from the `EffortLevel` enum's integer values via `EffortFromSliderIndex`/`EffortToSliderIndex` — `XHigh`/`Ultracode` are appended after `Max` in the enum (so legacy persisted values `Auto=0..Max=4` stay stable) but reordered for display; legacy `Auto` falls back to the High stop. ## Custom CLI Paths (CliPaths.cs)

- **Purpose**: Per-provider override pointing each agent at a specific CLI executable instead of relying on PATH / the built-in native install location — for tools installed in non-standard locations

## Terminal I/O (TerminalIO.cs)

- **Paste mechanism**: Saves full clipboard state → sets text → pastes into the terminal → sends Enter → restores clipboard. For Command Prompt (conhost) the paste is **handle-targeted** (see below); for Windows Terminal it is Ctrl+Shift+V (`PasteViaCtrlShiftVAsync`)
- **Handle-targeted conhost paste** (issues #76, #82, #83, multi-instance): every Command Prompt (conhost) provider pastes via `PasteViaConhostPasteCommandAsync()` — `PostMessage(terminalHandle, WM_COMMAND, ID_CONSOLE_PASTE /*0xFFF1*/, 0)` posts conhost's own Edit→Paste command straight to the terminal window. This invokes conhost's clipboard paste **regardless of QuickEdit/mouse-input mode**, so it covers plain shells (Claude Code, Codex, Cursor) and TUIs that capture the mouse (Claude API-key sign-in, PI, Antigravity, Open Code) with no per-character keystroke flooding. **Why handle-targeted**: the earlier paths were all global input — plain `mouse_event` right-click, Shift+Right-click (Open Code), keyboard menu navigation (Antigravity/mouse-mode), Ctrl+Shift+V — which go to the foreground window / cursor position, so with two VS instances open (dev + F5 experimental, sharing one clipboard) the prompt leaked into the **wrong** instance's terminal. `WM_COMMAND` is addressed to a specific HWND, so the paste always lands in this instance's terminal. History: #76 first typed the prompt via per-character `WM_CHAR` for mouse-input mode, but TUIs answer each char with cursor-position queries → the duplicated `[Pasted text]` flood of #83 and PI's R-loop crash in #82
- **Mouse-input-mode probe**: `SendTextToTerminalAsync` still probes `IsTerminalInMouseInputMode()` once up-front (in AgentCompletion.cs — AttachConsole + `GetConsoleMode` on CONIN$, reusing the guarded console-attach machinery + `_consoleSnapshotLock` + std-handle hygiene; best-effort, false on failure) off the UI thread for conhost providers (`_wtTabBarHeight == 0`, excluding Open Code), but now only to **skip the pre-deselect right-click** (QuickEdit is off so there is no selection to clear and a right-click would just open the context menu). The paste mechanism itself is the same `WM_COMMAND` for all conhost regardless of the probe
- **Keystroke path**: the `WM_CHAR` keystroke send (`SendTextViaKeystrokesAsync`) is used only by the `DisableClipboardSend` send (non-PI conhost). PI never takes the keystroke path (UserInput.cs forces it onto the clipboard/`WM_COMMAND` paste, still writing large/disable-clipboard prompts to a temp file so the pasted reference stays short)
- **Clipboard retry**: Up to 10 retries with 100ms delay for `CLIPBRD_E_CANT_OPEN`
- **Enter key varies by provider**: `WM_CHAR` (Claude/OpenCode), `KEYDOWN/KEYUP` (WSL), double-Enter (Codex)
- **Chunked paste**: `SendTextToTerminalAsync()` splits text over `PasteChunkSize` (24 KB) into sequential pastes; each chunk waits a scaled post-paste budget (`500–800ms base + min(len/divisor, cap)`) so Enter (after the final chunk) never races a streaming paste. Provider paste + wait live in `TriggerPasteAndWaitAsync()`
- **Large prompts as file (opt-in)**: when `SendLargePromptsAsFile` and the prompt exceeds ~1 KB, `SendButton_Click` writes it to `%TEMP%\ClaudeCodeVS_Session\<guid>\prompt-<ts>.md` and sends `Read and follow: <path>` — bypasses the conhost INPUT_RECORD buffer (truncates ~1 KB regardless of chunking), keeps `Files attached:` intact. WSL gets a converted path. Issue #48
- **Clipboard verification**: `SetClipboardAndVerifyAsync()` sets then reads back the clipboard before each paste — guards against a clipboard manager (Win+V, Ditto, RDP) overwriting it mid-send. Tolerant compare (`ClipboardTextMatches`) normalizes line endings/trailing `\0\n`. Retries 3× with 150ms backoff; on persistent failure logs and pastes anyway (issue #59). Contention → enable `DisableClipboardSend` (issue #61)

## Theme (Theme.cs)

- **Two distinct colors**: `terminalPanel.BackColor` is the live WPF panel color; `_terminalAgentColor` is the color the embedded terminal/agent was launched with. Theme-change prompts compare these two — the panel can drift away from the agent color across multiple theme switches without restarting the agent
- **Agent color set only on launch**: `RecordTerminalAgentColor()` is called at the end of both successful embed paths in `StartEmbeddedTerminalAsync` (WT and CMD). It snapshots `terminalPanel.BackColor` and persists it as `LastAgentTerminalColorArgb`. Never updated by `UpdateTerminalTheme` (which only re-tints the panel)
- **Restart prompt skip conditions**: No prompt when (a) a forced theme is active (`SelectedThemePreference != Automatic`), (b) terminal isn't running, or (c) `terminalPanel.BackColor == _terminalAgentColor` — covers two-dark-themes-same-RGB and reverting to a previously-declined color
- **Re-entrancy guard**: `_isShowingThemeRestartPrompt` prevents a second "Theme Changed" `MessageBox` stacking on the first. WPF dispatcher keeps pumping during the modal, so a later debounce tick from another `VSColorTheme.ThemeChanged` event would otherwise open a second dialog
- **Debounce**: 500ms `DispatcherTimer` collapses rapid consecutive theme events into one prompt
- **Forced theme overrides**: `ApplyForcedThemeResources()` injects 9 VS brush keys into the control's `Resources` dictionary so child WPF elements pick up the forced palette via `DynamicResource`. Removed when reverting to Automatic
- **Custom background color** (`ThemePreference.Custom` + `CustomThemeColorArgb`, default #F4ECFF): `GetCustomThemeColor()` returns the color; `ApplyForcedThemeResources()` derives fg (black/white) + hover/border shades from its brightness. Set in Settings → Theme via hex box + `ColorDialog`
- **Console colors written for every theme** (`SaveAndSetConsoleColorsRegistry`): the conhost background is set from `GetTerminalBackgroundColor()` (the current theme's bg — custom/dark/light/automatic alike) by writing `ColorTable00` (BGR) **explicitly on every launch**, `ColorTable07`=black/white by brightness, `ScreenColors`/`PopupColors`=`0x07` (bg index 0, fg index 7). Earlier code left `ColorTable00` untouched for non-custom themes and relied on the registry's existing value, so a leftover custom color stuck after switching back to Automatic/Dark/Light. Two guards prevent the custom color from being baked in permanently: (a) the explicit `ColorTable00` write overrides any stale value at launch, and (b) if `_consoleColorsSaved` is still true on entry (a prior launch saved originals but a superseded/aborted relaunch — issue #73 churn — skipped the restore), `RestoreConsoleColorsRegistry()` runs first so the *true* user originals are recovered before re-saving, instead of capturing the temporary themed color as the "original" that every later restore would write back. **Light-theme accent palette (issue #80)**: writing only `ColorTable00`/`07` left the 14 accent slots (`ColorTable01-06`, `08-15`) at their dark-tuned defaults, so on a light background the agent's cyan/blue tokens washed out. `WriteThemedConsolePalette()` now, when `Brightness(bg) > 150`, also paints a darker/saturated full palette from `GetLightConsolePalette()` (bright slots 09-14 mirror their normal counterparts so bright accents stay legible). All 16 slots are saved into `_savedConsoleColorTable[]` and restored (dark themes still leave the accent slots untouched)
- **Opt-out**: `SkipThemeRestartPrompt` setting (exposed in the consolidated Settings dialog) short-circuits `OnThemeChangeDebounceTimerTick` entirely — for users who auto-swap themes mid-session (e.g. VS debugging theme on F5) and don't want to be asked

## Session History (SessionHistory.cs)

- **Scope**: Claude Code (native) and Claude Code (WSL) only — other providers don't use this transcript format
- **Surfacing**: `SessionHistoryViewMenuItem` (in the `☰` Tools dropdown) or, if promoted, `SessionHistoryToolbarButton` — always shown now (issue #97); `ShowSessionHistoryDialogAsync` shows a "Claude Code only" message for other providers. Placement is handled by `RefreshToolbarLayout()` (`RefreshSessionHistoryButton()` delegates there)
- **Path encoding**: `EncodeClaudeProjectPath()` replicates Claude Code's filesystem encoding: every non-alphanumeric char → `-`. Example: `C:\Users\Daniel` → `C--Users-Daniel`
- **Directory resolution**: Native → `<config>\projects\<encoded-cwd>`, where `<config>` comes from `GetClaudeConfigDir()` — it honors the `CLAUDE_CONFIG_DIR` env var (the same one the CLI reads, supports `%VAR%` expansion) and falls back to `%USERPROFILE%\.claude` when unset/blank, so a relocated data store (e.g. on another drive) is found. WSL → shells out `wslpath -w "${CLAUDE_CONFIG_DIR:-$HOME/.claude}/projects/<encoded>"` (so a relocated config dir *inside* WSL is honored via the login shell's env) to get a `\\wsl.localhost\` UNC path readable by .NET. The WSL lookup drains stdout/stderr asynchronously and is bounded by a 5s timeout so a cold or hung WSL instance cannot leave the dialog loading forever
- **`GetClaudeConfigDir()`** is the shared `.claude`-root resolver (also reused by the native line-scroll keybinding installer in `Terminal.cs` so `keybindings.json` lands in the relocated config dir too)
- **JSONL parsing**: `ParseSessionFile()` reads with `FileShare.ReadWrite` (so the active session file isn't locked); extracts first user-typed message as preview, skips `tool_result`/`image` parts, sums `input_tokens + output_tokens` from assistant lines
- **Dialog**: WPF modal built programmatically; loads sessions async after open (shows "Loading…" placeholder); supports Resume (restart terminal with `--resume <uuid>`), Resume Last Session (`--continue`), Rename, Delete, Refresh. Layout (v35.0): a top filter bar (search box + "Renamed only" checkbox), the list, then a `DockPanel` button row with Refresh docked left and the launch/destructive actions grouped right. `CreateSessionItemContent()` builds each row as a `Grid` (left accent bar + stacked dim-meta line and title/preview line) and `BuildSessionTooltip()` the tooltip

## On Agent Finish (AgentCompletion.cs)

- **Scope**: any agent running in the **Command Prompt (conhost)** terminal — detection reads the console screen buffer, so it is provider-agnostic. **Windows Terminal supported (v41.0)**: the embedded WT window belongs to `WindowsTerminal.exe`, not the cmd.exe inside its ConPTY, so the watcher attaches to the **ConPTY console client resolved at launch** (`_wtConsoleClientPid`) and reads the same real screen buffer — alternate-screen TUIs (e.g. Devin) are detected and the read is cursor-blink independent. If that client can't be resolved/read, the watcher falls back to the UI Automation path (`_watchViaUia` / `TryCaptureWindowsTerminalText`, WT's UIA Text pattern). Default disabled. Configured in its own window (`ShowAgentFinishSettingsDialogAsync`, AgentFinishDialog.cs) opened from a button in the consolidated Settings dialog. **UI**: the "On Agent Finish…" button stays enabled for both terminal types; under WT an informational hint is shown (`SyncAgentFinishAvailability()`, wired to the terminal-type radios like `SyncDisableClipboardAvailability()`)
- **WT ConPTY client resolution**: just before launching WT, `StartEmbeddedTerminalAsync` snapshots existing cmd.exe PIDs (`SnapshotCmdProcessIds()`) and resets `_wtConsoleClientPid = 0`; after the WT window is embedded it calls `ResolveWtConsoleClientPid(preExistingCmdPids, terminalHandle)` (short retry loop). **Window-scoped match (fix, v50.0)**: it first resolves the new WT window's owning process via `GetWindowThreadProcessId(terminalHandle)` and looks for the new cmd.exe among *that process's* descendants only — this is what ties the result to the exact terminal just embedded. Only when that HWND isn't available/alive does it fall back to the old unscoped scan (new cmd.exe that is a descendant of *any* `WindowsTerminal.exe`), or the sole new cmd.exe if the tree walk lags. The unscoped scan alone was the bug: with two VS instances (or a fast restart before the old shell exits) launching WT around the same time, `newCmd` can hold more than one candidate and the old code returned the first `HashSet` match regardless of which window it belonged to — silently attaching the watcher to a different session's console, so detection read the wrong buffer and never lined up ("Windows Terminal detection not working properly"). Reset to 0 in `StopExistingTerminalAsync`. `TryCaptureConsoleText`/`TryCaptureConsoleHash` take an optional `directClientPid` that bypasses `ResolveConsoleClientPid` (window-based) and attaches straight to this PID; the tick passes `_watchedClientPidDirect`
- **Global default + per-solution override**: the effective config is resolved by `GetEffectiveAgentFinish()` — returns `_settings.ProjectAgentFinish[<solution name>]` when the open solution (keyed by `.sln` name via `GetCurrentSolutionName()`) has an entry, else the global `_settings.AgentFinish`. The watcher captures the effective config into `_watchedAgentFinish` at arm time so a mid-turn solution switch can't swap it. The dialog edits in-memory working copies (`CloneAgentFinish`) and on OK writes the global back and upserts/removes the per-solution entry. **Mid-turn settings edit**: after the dialog saves, it calls `RefreshWatchedAgentFinishConfig()`, which — only when a watch is currently active — re-resolves `GetEffectiveAgentFinish()` into `_watchedAgentFinish` so the just-saved settings apply to the in-flight turn when it finishes (and stops the watcher if the feature was turned off). The fire path already reads `IdleSeconds`/the action from `_watchedAgentFinish` live, so the swap takes effect for the current turn. (Enabling the feature mid-turn when it was off at arm time stays a no-op — no baseline/prompt-time was captured — so it applies from the next prompt)
- **Console-attach leak guard** (critical): the watcher `AttachConsole`s **VS itself** each tick. If VS is left attached, the **next `conhost.exe` spawn inherits VS's console instead of creating its own window and the terminal renders blank** (issue #73: "Restart code agent" stayed blank until VS was reopened; also visible after switching solutions). Three guards: `ResetAgentCompletionWatcher()` stops the watcher + dismisses the info bar on `OnBeforeCloseSolution`, the workspace-changed branch of `OnWorkspaceDirectoryChangedAsync`, **and at the top of every `StartEmbeddedTerminalAsync()`** (covers the restart button, provider/model switches, theme restarts, session resume); `EnsureNoConsoleAttached()` calls `FreeConsole` under a bounded `_consoleSnapshotLock` (`Monitor.TryEnter` w/ timeout), run after `StopExistingTerminalAsync()` and at the start of `ExecuteAgentFinishActionAsync` (no-op when VS has no console); and the conhost `Process.Start` itself runs inside a bounded `_consoleSnapshotLock` acquire + `FreeConsole` (background thread, 5 s timeout), so an in-flight capture can never overlap CreateProcess and a stuck attachment is cleared immediately before the spawn
- **Std-handle restore** (critical, the v14.0 fix for the OAF restart repro): `FreeConsole` alone is NOT enough — `AttachConsole` **replaces devenv's standard handles** with handles into the agent's console and `FreeConsole` leaves them dangling. Once the agent's console dies (restart kill), every later `conhost.exe -- cmd.exe` spawn inherits the dead std-handle values and **exits within ~100 ms with code 0**, so the panel stayed blank until VS was reopened no matter what the guards freed (reporter's repro: only after an OAF notification had fired — i.e. only after at least one capture had run). Fix: `CaptureOriginalStdHandlesOnce()` snapshots devenv's true originals before the first attach ever (static, process-wide); `RestoreOriginalStdHandles()` (`SetStdHandle` ×3) runs after every capture's `FreeConsole`, inside `EnsureNoConsoleAttached()`, and right before the conhost `Process.Start` (which logs `stdHandlesWereDirty=` so a user's terminal-launch.log can confirm/rule out this cause)
- **Arming**: `ArmAgentCompletionWatcherAsync()` is fired (fire-and-forget) at the end of `SendButton_Click`. Stores the conhost PID (`cmdProcess.Id`), takes an initial screen snapshot and send time, then starts `_agentCompletionTimer` (1 s `DispatcherTimer`). For Claude Code it also records a token baseline (newest `*.jsonl` via `CountTranscriptTokens`) to enrich the notification — **not** used for detection. Re-arming resets state; `StopAgentCompletionTimer()` also runs from `CleanupResources()`
- **Console-client PID**: the terminal launches as `conhost.exe`, but `AttachConsole` needs a console *client*, so `ResolveConsoleClientPid()` derives the cmd.exe PID from `GetWindowThreadProcessId(terminalHandle)` (returns the client by Windows back-compat) and falls back to the conhost's first ToolHelp32 child. Resolved fresh each sample so a restart can't pin a dead PID
- **Completion detection** (`OnAgentCompletionTimerTick` → `TryCaptureConsoleText`): each tick resolves the client PID, then `SetConsoleCtrlHandler(NULL,TRUE)` (shield VS from a shared-console Ctrl+C) → `AttachConsole` → `CreateFile("CONOUT$")` → `ReadConsoleOutputCharacter` → `FreeConsole` → restore Ctrl+C (all under `_consoleSnapshotLock`). Visible text **plus cursor pos** is FNV-1a hashed (`ComputeStableHash`); a changed hash ⇒ agent working. Fires only when activity was seen this turn **and** the hash held unchanged ≥ `IdleSeconds` (default 5, clamped 2–120). Dead PID or 30-min cap disarms. **Waiting-for-input suppression**: a static y/n or selection prompt reads as idle, so the settled screen is classified by `LooksLikeAgentInputPrompt()` (`AgentPromptKeywords` + `❯`/numbered-menu over bottom ~18 lines); on a match the tick returns without firing/disarming (and sets `_awaitingAgentInputReply`), so it fires once the user answers. Conservative by design. **Numbered-list false-positive fix (v27.0)**: a numbered *menu* is recognized only together with the `❯`/`›`/`▶` selection cursor (`sawArrow && menuItems >= 1`); the earlier "two-plus numbered options ⇒ prompt" rule made a finished turn whose final answer ended in a numbered list read as a waiting prompt, so the watcher backed off and the notification never fired. Real Claude Code selection/permission boxes always render the cursor, so genuine prompts stay suppressed while list-ending answers complete. **Prose-keyword false-positive fix (v46.0)**: the broad phrases `do you want to` / `do you trust` / `allow this` / `allow command` were removed from `AgentPromptKeywords` for the same reason — real permission/trust/choice boxes that use them also draw the `❯` cursor + numbered options (already caught structurally), but a *finished* turn whose prose answer merely ended with "Do you want me to …?" matched the keyword and was mis-suppressed, so the notification "sometimes never fired." The keyword list now keeps only waiting-state markers that render **without** a selection cursor (`(y/n)`/`[y/n]`/`(yes/no)`/`press enter to continue`/`press any key`). **Classic-renderer cursor + footer fix (v52.0)**: Claude Code's **non-alternate-screen** renderer draws the selection cursor as a plain ASCII `>` (not `❯`/`›`/`▶`), so a menu like "Pick next move? > 1. … 2. … 3. …" had `sawArrow=false` and was mis-read as a finished turn (fired "Agent finished" + the configured action over a live prompt). Two additions: (a) `sawArrow` is now also set when a line matches `^>\s+\d+[\.\)]\s` (a `>` cursor sitting directly on a numbered option — conservative so a stray `>` in prose or the input line `> /model …` isn't counted), and `>` is added to the numbered-item cursor class; (b) the selection-menu footer strings `esc to cancel` / `enter to select` / `to navigate` (from Claude Code's "Enter to select · ↑/↓ to navigate · Esc to cancel" line) were added to `AgentPromptKeywords` — highly menu-specific, so they suppress the false completion even when the cursor glyph isn't recognized. **Input-reply backoff** (v14.0, reworked in v18.0): while `_awaitingAgentInputReply` is set, the capture must not fight the user as they answer — each `AttachConsole`/`FreeConsole` on VS bounces the embedded conhost's keyboard focus, which is what made arrow keys/typed answers to agent menus unreliable. v14.0 backed off from 1 s to `InputPromptRecheckMs` (10 s) but only `&& IsTerminalFocused()` — exactly wrong, because while the user clicked *into* an unfocused terminal to answer, the 1 s attach storm kept knocking focus back out (they had to click the panel and the agent tab repeatedly). v18.0 splits on focus the other way: **focused → never attach** (nothing to detect until the user acts, and a focused user can see the result, so a delayed finish notification costs nothing — this is what keeps a poll from knocking focus out mid-reply); **unfocused → attach but only every 10 s** (so a user clicking in isn't fought by a rapid storm, while an agent that resumes on its own is still noticed within ~10 s). Once unfocused-and-read shows the screen no longer looks like a prompt (user answered → agent working), `_awaitingAgentInputReply` clears and the normal 1 s cadence resumes. **Sticky waiting + early classification** (v15.0): the prompt classification now runs as soon as the settled screen holds still for one tick (before the full idle window), so the menu backs off the cadence within ~1 s of appearing instead of after `IdleSeconds`; and on a hash change `_awaitingAgentInputReply` is re-evaluated as `LooksLikeAgentInputPrompt(text)` rather than unconditionally cleared. Arrow-keying a selection menu moves the `❯` cursor and so changes the screen — clearing the flag on every such change dropped the cadence back to 1 s mid-navigation and the next attach ate the arrow keys (reported in plan mode). Keeping the flag set while the changed screen still looks like a prompt holds the 10 s cadence throughout navigation; it clears (resuming 1 s) only once the screen no longer looks like a prompt (user answered → agent working). **Typing-interference guard**: the capture briefly attaches VS (disturbs keystrokes), so each tick skips the read while the user is *actively typing* — gated on a recent terminal keystroke (`_lastTerminalKeyUtc`, within `TerminalTypingGuardMs`=1500) **and** `IsTerminalFocused()`. Mere focus isn't enough (pasting leaves the terminal focused but un-typed — a focus-only gate stalled the turn ~15s). **Prompt-typing guard (v27.0)**: the tick also skips while the user recently typed in the WPF prompt box (`_lastPromptKeyUtc`, set in `PromptTextBox_PreviewKeyDown`, within the same window) — the attach was bouncing keyboard focus out of the prompt mid-keystroke during active generation ("sometimes I can't type while the agent works"). No focus check needed here: a recent prompt keystroke already proves the user is in the prompt, and 1.5s after they stop, detection resumes. **Focus-restore around the read (v30.0)**: the typing guards only pause the poll while the user is *actively* typing (a keystroke within `TerminalTypingGuardMs`). When the user pauses longer than that, the next tick's `AttachConsole`/`FreeConsole` still bounces native keyboard focus off the embedded terminal (or the WPF prompt), and the now-unfocused terminal swallows the next keystroke — which, because it never reaches the terminal, can't refresh `_lastTerminalKeyUtc` to re-arm the guard, so the poll keeps stealing focus every second and the terminal stays dead for the rest of the turn (the recurring "console stops accepting keyboard input after a while", reported with OAF enabled on Command Prompt). The tick now snapshots `GetFocus()` immediately **before** the `Task.Run` read and, if it was one of our input surfaces (`IsOwnedInputFocusWindow()` — the terminal/its child, or the WPF `HwndSource` host), restores it with `SetFocus` right after the read when it moved. Captured on the UI thread (joined to the terminal's input queue via SetParent, so `GetFocus`/`SetFocus` see the terminal the same way the focus guards do); restore runs before any of the post-read early-returns so every read path is covered. Makes the poll focus-neutral rather than relying solely on the active-typing skip. **Selection-freeze guard (v34.0)**: clicking or marking text in a QuickEdit conhost puts it into selection/mark mode, which **freezes the screen buffer** — the agent keeps working but the captured text stops changing, so the watcher read a static screen and fired a premature "finished" notification the moment the user clicked into the terminal mid-turn (issue #94). Each capture now calls `GetConsoleSelectionInfo` while attached and sets `_consoleSelectionActive` when `CONSOLE_SELECTION_IN_PROGRESS`/`CONSOLE_SELECTION_NOT_EMPTY`/`CONSOLE_MOUSE_DOWN` is set; the tick treats that as activity (pushes `_lastConsoleChangeUtc` forward and returns without firing), so detection resumes only once the user clears the selection and conhost unfreezes the buffer. **Question sound (v52.0)**: `AgentFinishConfig.PlayQuestionSound` (opt-in, independent of `PlaySound`) plays a **distinct descending** tone (`PlayQuestionSound()`, C6→G5→D5) when the agent parks at a prompt instead of finishing, so "the agent needs me" is audibly distinct from the rising finish chime. Fired via `MaybePlayQuestionSound()` from `EvaluateScreenQuiescenceAsync` — at the false→true transition in the hash-changed branch and again in the settle branch — guarded by `_questionSoundPlayed` so it sounds **once per waiting episode**; the guard clears when a later screen no longer looks like a prompt (agent resumed), so the next question sounds again. Reset (with `_awaitingAgentInputReply`) on each arm
- **Modal-dialog guard** (v29.0, hang fix): `OnAgentCompletionTimerTick` skips the entire tick when `System.Windows.Interop.ComponentDispatcher.IsThreadModal` is true (and pushes `_lastConsoleChangeUtc` forward so the turn is detected as finished only after the dialog closes). The watcher rides a `DispatcherTimer`, which **keeps ticking inside the nested modal message loop** that an open WPF dialog pumps — so with a turn settling while the consolidated Settings dialog (or the On-Agent-Finish config sub-dialog, or a `MessageBox`) was open, the tick ran the process-global `AttachConsole`/`FreeConsole` read and could fire the action (`dte.ExecuteCommand("Debug.Start"/"Build…")`) or add the info bar to the **modally-disabled** main window from underneath that loop, deadlocking Visual Studio. `IsThreadModal` is true on the UI thread for any WPF modal, so the single check covers every dialog with no per-call-site wiring
- **Notify**: `OnAgentTurnCompletedAsync` plays a chime (opt-in) and shows a VS **main-window info bar** (`ShowAgentFinishNotificationAsync` via `__VSSPROPID7.VSSPROPID_MainWindowInfoBarHost` + `SVsInfoBarUIFactory`) reading `Agent finished · <duration>` (+`<Δtokens>` only for Claude), shown even when the tool window is hidden. `AgentFinishInfoBarEvents` (`IVsInfoBarUIEvents`) handles close/unadvise and the action hyperlink
- **Actions** (`ExecuteMainActionAsync`, returns `bool` success): Build/Rebuild → `BuildAndCheckSuccessAsync()` (`dte.ExecuteCommand` + `WaitForDteBuildToFinishAsync` + `SolutionBuild.LastBuildInfo` — false on timeout or a failed project); Run/RunWithoutDebugging → `PrepareAndRunAsync()` (also returns `bool`: false if the clean/rebuild gate failed or timed out, otherwise true after issuing `Debug.Start`/`Debug.StartWithoutDebugging`); RunTests → `dte.ExecuteCommand("TestExplorer.RunAllTests")` (fire-and-forget, always true — no completion signal available); RunScript → `RunFinishScriptAsync` `Process.Start` (`UseShellExecute`) in the workspace dir, always true (fire-and-forget) — a `.cmd`/`.bat` is wrapped as `cmd.exe /k "<path>"` and a `.ps1` as `powershell.exe -ExecutionPolicy Bypass -NoExit -File "<path>"` (the `.ps1` default shell verb is Edit, so shell-executing it directly would open an editor; `/k`/`-NoExit` keep the console open for output), anything else is shell-executed directly; SendToAgent → `SendTextToTerminalAsync`, always true. `RequireFileChanges` gates on `git status --porcelain` in `_gitRepositoryRoot` (non-git ⇒ never blocks); the change check runs **before** the chime/info-bar in `OnAgentTurnCompletedAsync`, so when the action is skipped for lack of changes the turn is fully silent (no sound, no notification) rather than announcing the skip. `Confirm` (default true) surfaces the action as an info-bar button instead of auto-running
- **Follow-up ("also send to agent")**: `AgentFinishConfig.FollowUpSendToAgent` (optional) sends a second literal text to the agent — e.g. "Commit and push the changes" — after `Action` succeeds. `HasFollowUp()` gates it on a non-empty value **and** `Action` being neither `None` (nothing ran) nor `SendToAgent` (already sends text itself; a second send has no success signal to key off and could race the agent's own next turn). UI field in `ShowAgentFinishSettingsDialogAsync` is enabled/disabled per-action via `SyncFollowUpOptions()`. **Non-confirm path**: `ExecuteAgentFinishActionAsync()` runs `ExecuteMainActionAsync` then, only if it returned true, sends the follow-up — one uninterrupted auto-run. **Confirm path**: `OnAgentTurnCompletedAsync` does **not** chain the follow-up behind the main action's button — it shows a first info bar/button for the main action, and only after that's clicked and `ExecuteMainActionAsync` returns true does it show a **second**, separate info bar/button (`DescribeSendCommand`-labeled) for the follow-up, so a build failure (or a declined/skipped action) never silently triggers the commit-and-push step. **Presets (v50.0)**: a "Preset ▾" `Button` + `ContextMenu` next to the follow-up `TextBox` (`FollowUpPresets`, an array of `(label, text)` tuples in AgentFinishDialog.cs) lets the user fill the field from canned text instead of typing it — currently a plain "Commit and push the changes" and a "no AI credit" variant that asks the agent to keep only the human author and drop AI references from the commit. Picking an item overwrites the field (a follow-up is a single message, not something to append to). Add more entries to `FollowUpPresets` to offer more canned follow-ups
- **Notification label length (v50.0)**: `DescribeSendCommand()` (used for both the `SendToAgent` action's button label and the follow-up's second-notification label) truncates at `SendCommandLabelMaxChars` (100, was 24) — the VS info-bar hyperlink can render much more than a button-sized label, and 24 chars cut off canned follow-ups like the presets above well before the meaningful part
- **Console interop** (Interop.cs): reuses `AttachConsole`/`FreeConsole`/`CloseHandle`/`COORD`; adds `CreateFile`, `GetConsoleScreenBufferInfo`, `ReadConsoleOutputCharacterW`, `SetConsoleCtrlHandler`, `Beep`, and `SMALL_RECT`/`CONSOLE_SCREEN_BUFFER_INFO`; `GetConsoleMode` + `ENABLE_QUICK_EDIT_MODE` (0x40) back `IsTerminalInMouseInputMode()`, the paste/zoom QuickEdit probe for issue #76/#78 (reads CONIN$ mode under the same guarded attach as the screen capture); `GetConsoleSelectionInfo` + `CONSOLE_SELECTION_INFO` back the v34.0 selection-freeze guard (issue #94 — see Completion detection above); `GetCurrentConsoleFontEx`/`SetCurrentConsoleFontEx` + `CONSOLE_FONT_INFOEX` back `TryAdjustConhostFontSize()` (changes the conhost cell height on CONOUT$ for the input-mode-independent Ctrl+Scroll zoom, issue #78)

## "@" File/Folder Picker (AtMention.cs)

- **Trigger**: `PromptTextBox`'s `TextChanged` (wired in XAML) calls `UpdateAtMentionPopup()`, which detects an `@` token under the caret (`@` at text start or after whitespace, followed by non-whitespace). Always on; no setting
- **Index**: `EnumerateWorkspaceEntries()` walks `GetWorkspaceDirectoryAsync()` on a background thread, skipping build/VCS/package dirs (`AtIgnoredDirs`) and reparse points, returning workspace-relative `/`-separated paths (folders carry a trailing `/`), capped at 8000. Cached per-root with a 30 s TTL (`EnsureAtEntriesAsync`); a stale index refreshes in the background while current results still show. First trigger shows an "Indexing…" row, then `EnsureThenRefilterAsync` re-runs the filter
- **Popup**: a programmatic `Popup` + `ListBox` (`EnsureAtPopup`), positioned at the caret via `GetRectFromCharacterIndex`. Themed with `GetThemeBrushes` + a derived hover brush (`ComputeAtHoverBrush`) so selected/hover rows stay readable in a standalone popup (no system-blue). Closes on real focus loss but not when focus enters the popup (`IsInsideAtPopup`), so a mouse click can commit first
- **Keys**: `HandleAtMentionKey()` runs at the top of `PromptTextBox_PreviewKeyDown` (before history nav / send-on-Enter) and consumes Up/Down/Enter/Tab/Esc while the popup is open
- **Ranking** (`RankAtEntries`): a query may contain `/` for folder drill-down — the part after the last `/` matches the entry name and the prefix constrains the subtree; name prefix-matches rank above name/path substring matches
- **Insert** (`CommitAtSelection`): replaces the typed `@query` with `@<relative-path>`; a file appends a space and closes, a folder leaves the caret in place and re-opens the picker to drill in. Relative paths resolve for every provider (terminal cwd = workspace), so no WSL conversion. `_atSuppressTextChanged` guards the programmatic edit
